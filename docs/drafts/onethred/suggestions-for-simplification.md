# 对 onethred 方案的简化建议

## 总体评价

方案设计全面、考虑周到，但可能**过度设计**（over-engineered）。对于50-200站的规模，可以更简洁。

## 主要建议

### 1. 精简文档结构

**当前**：16个章节，539行  
**建议**：合并为8个核心章节，删除"阶段C超大规模"和部分重复内容

```text
1. 问题与目标（保留）
2. 总体架构（精简）
3. 配置与发现（保留）
4. 运行时行为（精简）
5. CLI与systemd（合并）
6. 迁移方案（保留）
7. 测试与验收（精简）
8. 风险与决策（合并）
```

### 2. 推迟过早优化

**删除或移至"后续考虑"的内容**：
- 阶段C的多机分片、SQLite
- `MONITOR_GLOBAL_MAX_INFLIGHT` 全局并发信号量
- `/health` 状态文件（阶段B）
- 线程崩溃自愈的复杂逻辑

**理由**：先跑起来，遇到实际瓶颈再优化

### 3. 简化错误处理

**当前**：详细的错误分类和恢复策略  
**建议**：第一版采用简单策略

```text
- 401：refresh一次 → login一次 → 失败则退避
- 其他错误：保留token，退避重试
- 禁止：因一站失败退出整个进程
```

### 4. 重新评估错峰启动

**问题**：是否真的需要错峰？  
**分析**：
- 300秒周期内100站同时启动，平均间隔3秒
- 实际上不太可能触发限流
- 增加了实现复杂度

**建议**：第一版不做错峰，观察实际影响

### 5. 明确MVP（最小可行产品）

```text
MVP功能：
✅ 单进程加载多站配置
✅ 每站一线程运行monitor_loop
✅ 统一SIGTERM停止
✅ --sites-dir和--env-file互斥
✅ 基础错误处理（不退出进程）

非MVP（先不做）：
❌ 错峰启动
❌ 线程崩溃自愈
❌ 热加载
❌ 状态文件
❌ 全局并发控制
```

### 6. 简化systemd配置

**当前**：详细的security选项  
**建议**：第一版只保留必要配置

```ini
[Service]
Type=simple
ExecStart=...
Restart=always
RestartSec=10
TimeoutStopSec=60
```

安全加固可以后续添加。

### 7. 减少配置选项

**删除或硬编码**：
- `MONITOR_STAGGER_STEP_SECONDS`（暂不需要）
- `MONITOR_MAX_SITE_THREADS`（暂不需要）
- `MONITOR_WORKER_RESTART_LIMIT`（简化）

**保留**：
- `MONITOR_SITES_DIR`（默认sites）
- `LOG_LEVEL`

### 8. 简化测试要求

**当前**：详细的测试矩阵  
**建议**：MVP测试重点

```text
必须测试：
- 两站并行运行
- 一站失败不影响另一站
- SIGTERM优雅退出
- 配置校验

可后续补充：
- 错峰启动验证
- 线程崩溃恢复
- 路径冲突检测
```

## 实施优先级

```text
第一周：MVP核心功能
第二周：真实环境测试
第三周：根据实际问题添加必要功能
```

## 预期收益

1. **开发时间**：减少40-50%
2. **代码复杂度**：显著降低
3. **维护成本**：初期更低
4. **风险**：更小（功能更少）

## 结论

方案基础很好，但可以更**渐进式**发展。先用最简单的方式解决核心问题（内存和systemd数量），遇到实际瓶颈再优化。避免"为可能永远不会出现的问题设计解决方案"。
