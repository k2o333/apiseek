# 资源与容量模型（单进程多线程）

> 配合 `sub2api-multithread-multi-site.md`  
> 数字为量级估算 + 本机实测锚点，落地后应以实测定标。

---

## 1. 实测锚点（本机，一站一进程）

| 项目 | 值 |
|------|-----|
| 单站 RSS | ~32–33 MB |
| 单站 PSS | ~24 MB |
| 线程数 | 1 |
| 空闲 CPU | ~0% |
| 轮询周期 | 300s 默认 |

两站两进程合计 RSS ≈ **65 MB**。

---

## 2. 成本拆解

```text
总内存 ≈ 解释器基线 + requests/SSL 缓存 + Σ(每站状态)
```

| 组成部分 | 多进程 | 单进程多线程 |
|----------|--------|--------------|
| CPython + 标准库 + site-packages | **每进程一份** ~20–30MB | **全进程一份** |
| 每站 Config / Token / 分组快照缓存 | 小（KB～百 KB） | 同左 |
| 每站 requests Session | 有 | 有 |
| 每线程栈（Linux 默认可达 8MB 虚拟） | 1 | N；**RSS 通常远小于 N×8MB**（按需提交） |

因此站数从 2 → 100 时：

- 多进程：近似 **线性 ×30MB**  
- 多线程：基线近似不变，**每站边际成本远小于 30MB**（常见数 MB 内，视连接与缓存）

---

## 3. 粗算表（规划用）

假设：

- 多进程：30 MB/站  
- 多线程：基线 40 MB + 2 MB/站（偏保守时可按 3～4 MB/站）

| 站数 N | 多进程 RSS | 多线程 RSS（估） | 说明 |
|--------|------------|------------------|------|
| 2 | 60 MB | 45–50 MB | 接近，优势不大 |
| 10 | 300 MB | 60–80 MB | 开始拉开 |
| 50 | 1.5 GB | 140–240 MB | 多进程已肉疼 |
| 100 | 3 GB | 240–440 MB | 多线程合理目标带 |
| 200 | 6 GB | 440–840 MB | 考虑线程池或分片 |

落地后用：

```bash
ps -o rss,vsz,nlwp,cmd -p $(systemctl show -p MainPID --value sub2api-monitor)
```

`nlwp` ≈ 1 + 站数（主线程 + worker）。

---

## 4. CPU

假设：

- 周期 T = 300s  
- 单站有效工作时间 w = 1～5s（login 偶发 + groups）  
- 站数 N = 100  

平均 CPU 占用（单核比例）粗算：

```text
avg ≈ (N * w) / T / cores
    ≈ (100 * 3) / 300 / 1 = 1.0  → 约 100% 的 1 秒… 摊到 300s 即 ~1% 单核
```

更直观：每 300s 总共约 300 秒站·工作时间 → 平均约 **1 个逻辑核的满载** 的 **1 秒/秒** 量级若 w=3 且完全串行；**并行** 时 wall 短、CPU 秒数类似，瞬时核数升高。

**结论：** CPU 通常不是瓶颈；**网络 RTT、TLS、源站限流** 才是。

---

## 5. 网络与限流

| 风险 | 表现 | 控制 |
|------|------|------|
| 启动惊群 | 同时 100 次 login | 错峰 STAGGER |
| 周期对齐 | 整点同时打 | jitter |
| 单 IP 被 Cloudflare/WAF | 403/429 | 代理池按站配置；降并发 |
| 全局并发过高 | 本地 ephemeral port / 连接表 | 可选全局 Semaphore |

可选参数（实现时）：

```text
MONITOR_GLOBAL_MAX_INFLIGHT=20   # 同时处于 HTTP 中的站数上限
```

与「每站一线程」兼容：线程在 Semaphore 外 sleep，进入 poll 前 acquire。

---

## 6. 文件描述符与线程

| 资源 | 估算 | 建议 |
|------|------|------|
| 线程 | 1 + N | N≤200 一般 OK；TasksMax=512 |
| FD | 每活跃 Session 少量 + 每站 2～3 个打开文件峰值 | LimitNOFILE=65535 |
| inode/写 | latest 原子替换 + 偶发 events append | 普通 SSD 无压力 |

---

## 7. 磁盘

| 文件 | 增长 |
|------|------|
| token.json | 近似常数 KB |
| groups_latest.json | 近似常数（完整快照） |
| groups_events.jsonl | 仅变化时追加；180 天裁剪 |

100 站 × 若每天多次变更，events 需监控磁盘；与线程模型无关。

---

## 8. 何时升级架构

| 信号 | 动作 |
|------|------|
| RSS > 机器内存 30% 且 N 大 | 降 per-site 缓存；或线程池；或按 shard 多进程 |
| 线程数 > 500 | 改调度池，不要 1 站 1 线程 |
| 单 IP 限流严重 | 代理、拉长 interval、分片多出口 |
| 需要跨站查询 | SQLite / 外部存储（独立议题） |

---

## 9. 容量验收建议

上线多线程版本后固定记录：

```text
N 站、RSS、PSS、nlwp、一次 --once 墙钟时间、错误率
```

对比迁移前「N 个 @ 实例」的 `ps` 汇总，作为优化成功判据。
